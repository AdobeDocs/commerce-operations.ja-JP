# Adobe Commerce用語

## A

### アバブザフォールド

_形容詞_

ブラウザーウィンドウで、web ページの読み込み直後とユーザーがページを下にスクロールする前に表示されるコンテンツ。
レイアウトをデザインするときは、柔軟な形式を使用して、優先順位の高い製品、機能、販売、通知、オプションなどを最適に表示します。

モバイルとタブレットでは、画面のサイズやサイズ、画面の向き（ポートレートと横向き）など、折り目の上の領域が大きく異なります。
レスポンシブ機能を利用したテーマとテストは、コンテンツとレイアウトの適切な組み合わせを判断するのに役立ちます。

_Term属性&#x200B;:_

* _フィールド：デザイン_

### アクティブな分岐

_名詞_

アクティブなブランチまたは環境とは、サービスにアクセスできるデプロイ済みインスタンスまたは実行中のインスタンスに接続されたブランチまたは環境のことです。
非アクティブ化すると、サービスと実行中のインスタンスへの接続は削除されますが、コードは保持されます。
普通のGit ブランチになります。

_Term属性&#x200B;:_

* _フィールド：cloud_

### アダプタ

_名詞_

システムのソースコードを変更することなく、他の点では互換性のない2つのシステムを連携させるクラス。
例としては、データベース アダプタ、キャッシュ アダプタ、ファイル システム アダプタ、ポストプロセッサ ライブラリ アダプタ、およびその他の種類のコンピューティング アダプタがあります。

_Term属性&#x200B;:_

* _フィールド：プログラミング_

### 管理者

_名詞_

ソフトウェアでは、すべての機能を管理するための完全な管理者権限を持つユーザーロール。
Adobe Commerceでは、管理者ユーザーは管理者のすべての権限を持ち、管理者のすべての機能、オプション、および機能にアクセスできます。
また、ユーザーや役割を作成することもできます。

詳細：[&#x200B; ユーザーの追加](https://experienceleague.adobe.com/docs/commerce-admin/systems/user-accounts/permissions-users-all.html?lang=ja)

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア_
* _同義語：管理者、スーパーユーザー_
* _関連用語：コマース管理者_

### 管理領域

_名詞_

注文、カタログ、コンテンツ、設定を管理する、ストアのパスワードで保護されたバックオフィス。
管理画面にアクセスすると、商品、注文、配送、CMSのコンテンツ、ストアフロントのデザイン、お客様情報など、ストアを管理できます。
管理者ユーザーには、機能、オプション、および機能へのアクセスを制御する権限を持つ関連する役割があります。

詳細：[Adobe Commerce ユーザーガイド &#x200B;](https://experienceleague.adobe.com/docs/commerce-admin/user-guides/home.html?lang=ja)

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア_
* _類義語：管理者、管理パネル、バックエンド、管理インターフェイス、管理UI_
* _関連する用語：管理者_

### 管理者変数

_名詞_

管理者変数は、管理者ユーザーアカウントが管理者UIにアクセスするための設定設定を上書きするためのプロジェクト環境変数です。

詳細：[管理者変数](https://experienceleague.adobe.com/docs/commerce-cloud-service/user-guide/configure/env/stage/variables-admin.html?lang=ja)

_Term属性&#x200B;:_

* _フィールド：管理者、クラウド_

### adminhtml

_名詞_

管理者に割り当てられた内部領域名。

詳細：[Adobe Commerce ユーザーガイド &#x200B;](https://experienceleague.adobe.com/docs/commerce-admin/user-guides/home.html?lang=ja)

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア_
* _関連用語：エリア、コマース管理者_

### エリア

_名詞_

「領域」は、Magentoのアプリケーションスコープの抽象的な用語です。
領域は、最適化されたリクエスト処理のためにコードを整理する論理コンポーネントです。
領域は、ストアフロントからアクセスする設定オブジェクトのメモリ要求を減らし、必要な依存コードのみを読み込むことでweb サービス呼び出しを効率化します。
各エリアには、URLとリクエストを処理するための完全に異なるコードを含めることができます。

Adobe Commerceの主な領域：

* 管理者（adminhtml）
* ストアフロント
* Web API REST （webapi_rest）

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア_
* _関連用語：コマースコンポーネント、ストアフロント_

### 属性

_名詞_

製品の特徴または特性で、製品の何らかの側面を記述する。
Adobe Commerce ユーザーは、カスタム属性を作成して、デフォルトの属性セットまたはカスタム属性セットに追加できます。
これらの属性は、管理者またはプログラムで作成します。
例：色、サイズ、重量、価格、年齢、性別など

カスタム属性は、EAV （Entity-Attribute-Value）属性の一種です。

Google Shopping ads ChannelやAmazon Sales Channelなどの統合機能では、Commerceの属性をサードパーティの属性にマッピングし、商品の適切な表示や販売、広告の表示を行うことができます。

詳細：[EAVおよびextension_attributes](https://developer.adobe.com/commerce/php/development/components/attributes/)

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア、製品_
* _類義語：製品属性、カスタム属性_
* _関連用語：拡張属性_

属性セット内の属性の論理グループ。

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア_

_名詞_

特定の製品向けにカスタマイズされた属性グループのコレクション。

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア、製品_
* _関連用語：属性_

### 平均在庫コスト

_名詞_

商品価格、クーポン、割引に加えて、運賃と該当する税金が加算されます。
平均は、各月の在庫の開始コストと、期間の最後の月の在庫の終了コストを加えて決定されます。

_Term属性&#x200B;:_

* _フィールド：製品、価格、在庫_

## B

### 基本通貨

_名詞_

すべてのオンライン決済にストアビューごとに使用されるプライマリ通貨。
店舗は、世界中の200 カ国以上の通貨を受け入れることができます。
ストアフロントでは、特定の国やロケール向けに、複数の通貨を選択できる通貨セレクターを提供しています。
通貨記号は、商品の価格や、注文や請求書などの販売文書に表示されます。
必要に応じて通貨記号をカスタマイズし、店舗またはビューごとに価格の表示を個別に設定できます。

詳細：[通貨](https://experienceleague.adobe.com/docs/commerce-admin/stores-sales/site-store/currency/currency.html?lang=ja)

* _フィールド：価格_

### ブロック

_名詞_

特定のコンテンツ（情報、ユーザーインターフェイス要素）をレンダリングする、ページ出力の単位です。エンドユーザーにとって視覚的に有形なものです。
[&#x200B; ブロック &#x200B;](https://experienceleague.adobe.com/docs/commerce-admin/content-design/elements/blocks/blocks.html?lang=ja)は実装され、モジュールによって提供されます。
ブロックは、テンプレートを使用してHTMLを生成します。
ブロックの例には、カテゴリーリスト、ミニカート、製品タグ、製品リストなどがあります。

[動的ブロック &#x200B;](https://experienceleague.adobe.com/docs/commerce-admin/content-design/elements/dynamic-blocks/dynamic-blocks.html?lang=ja)は、価格ルールなどのロジックに基づいてコンテンツを提供します。

ページビルダーは、[&#x200B; ブロック &#x200B;](https://experienceleague.adobe.com/docs/commerce-admin/page-builder/add-content/block.html?lang=ja)と[動的ブロック &#x200B;](https://experienceleague.adobe.com/docs/commerce-admin/page-builder/add-content/dynamic-block.html?lang=ja)のインタラクティブな操作と作成を拡張します。

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア_
* _類義語：動的ブロック_
* _関連用語：cms ブロック、静的ブロック、コンテナ、レイアウト_

### ブランド

_名詞、動詞_

製造元またはDesignerによって特定の商品または商品グループを定義する一意のID。
これらには、衣料品、家電製品、高級品などのブランド名が含まれます。
また、事業部門やターゲットオーディエンスなどに基づいて、複数のブランドで製品を販売する企業もあります。

カスタム属性を使用して、商品のブランド情報を保存します。

拡張機能や統合機能の中には、Google Shopping Ads ChannelやAmazon Sales Channelなど、商品にブランドを使用したり、ブランドを必要とするものもあります。

_Term属性&#x200B;:_

* _フィールド：ビジネス_

### カタログ

_名詞_

マーチャントの場合、カタログは商品の在庫を表します。
Adobe Commerceのアーキテクチャでは、カタログはカテゴリー、商品、商品属性で構成されています。

Commerceの各ストアには1つの商品カタログしかなく、すべてのストアビューで共有されています。
マルチストアのインストールでは、各ストアで個別のカタログを持つことも、カタログを共有することもできます。
現在のストアカタログは、デフォルトのルートカテゴリによって決まり、ストアの上部ナビゲーション（メインメニュー）でユーザーに表示されます。
（「ルートカテゴリ」という用語は、実際にはカテゴリではなく、カテゴリとサブカテゴリで構成されるメニューのコンテナであるため、混乱を招く可能性があります）。

必要な数のルートカテゴリを作成できますが、一度に使用できるのは1つ（デフォルト）のみです。

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア、価格設定、製品_
* _関連用語：共有カタログ、カタログルール_

### コンフィグ商品

_名詞_

設定可能な製品は、各バリエーションのオプションをドロップダウンリストに示す単一の製品のように見えます。
各オプションは、実際には独自のSKUを持つ独立したシンプルな商品であり、商品バリエーションごとに在庫を追跡することが可能です。
同様の効果を得るには、カスタムオプションでシンプルな商品を使用できますが、各バリエーションの在庫を追跡する機能はありません。
複数のオプションを持つ製品は、複合製品と呼ばれることもあります。

設定可能な製品では、より多くのSKUを使用し、最初は設定に少し時間がかかることがありますが、最終的には時間を節約できます。
ビジネスを拡大したい場合は、複数の選択肢がある製品であれば、設定可能な製品タイプの方が適しているかもしれません。

例：2色と3 サイズのT シャツ。
6つのバリエーションを個別の製品として作成する必要があります（それぞれに独自のSKUがあります）。
あらゆるバリエーションが設定可能な商品に追加され、顧客がサイズと色を選択してカートに追加します。

_Term属性&#x200B;:_

* _フィールド：コマースソフトウェア、製品_
* _関連用語：製品タイプ_

* カスタマーサービスの負担を軽減し、応答性を向上
* 顧客体験とロイヤルティの向上

### 上方

_名詞_

[PWA Studio](https://github.com/magento/pwa-studio)は、開発中に[上向き](https://developer.adobe.com/commerce/pwa-studio/guides/packages/upward/)を使用しています。
UPWARDは、Unified Progressive Web App Response Definitionの頭字語です。
UPWARD定義ファイルは、web サーバーがProgressive Web Applicationを配信およびサポートする方法を記述します。

上方定義ファイルは、プラットフォームに依存しない宣言言語を使用したサーバーの動作に関する詳細を提供します。
これにより、アプリケーションはUPWARD サーバーからのHTTP エンドポイントの動作のみを懸念するため、Progressive Web Applicationを任意のテクノロジースタック上の任意の言語でUPWARD準拠のサーバー上で実行できます。

UPWARD サーバーは、定義ファイルを使用して、アプリケーションシェルからのリクエストに対する適切なプロセスまたはサービスを決定します。
これは、サーバーがリクエストを処理し、その応答を構築する方法を説明します。

PWA プロジェクトには、サービスの依存関係を指定するためのUPWARD定義ファイルを含めることができます。

_Term属性&#x200B;:_

* _フィールド：デザイン、コマースソフトウェア、プログラミング_
* _同義語：PWA Studio、統合プログレッシブ Web アプリ応答の定義_
