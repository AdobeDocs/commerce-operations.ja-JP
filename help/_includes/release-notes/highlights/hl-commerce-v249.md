# Adobe Commerceのハイライト（v2.4.9）

## v2.4.9のハイライト

Adobe Commerce 2.4.9 リリースには、次のハイライトが適用されます。

### REST APIの変更

#### ストアビューレベルでのREST API製品ギャラリーの継承コントロール

ストアスコープでREST APIを使用して製品を更新すると、`media_gallery_entries`がペイロードから省略されたり、`NULL`に設定されたりすると、製品画像やビデオがグローバルスコープから変更を継承しなくなります。 対応するフィールドを`NULL`に設定することで、REST APIを介して製品画像とビデオのスコープ継承を復元できるようになりました。

ストアビューレベルでREST APIを使用して製品を更新する場合：

- `media_gallery_entries`を省略するかNULLに設定すると、デフォルト/グローバルギャラリーからの継承が保持されるようになりました。
- 特定のギャラリー属性（`label`、`position`、`disabled`、`video_title`、`video_description`など）の継承を復元するには、これらのフィールドを`media_gallery_entries`配列内のNULLに設定します。

この変更により、ストアビューでデフォルトのギャラリー値を簡単に維持または復元できるようになり、混乱が軽減され、ギャラリー管理の一貫性が高まります。

_ACP2E-4358 - [GitHub コードの貢献度](https://github.com/magento/magento2/commit/f7bbcb4e)_

### GraphQL APIの変更

#### Magento Open Sourceで`clearCart`個のGraphQL変異を利用できるようになりました

[clearCart](https://developer.adobe.com/commerce/webapi/graphql/schema/cart/mutations/clear-cart/) GraphQLの変異は、すべてのMagento Open Source ユーザーが利用できるようになりました。 以前は、この変異はAdobe Commerceでのみアクセス可能でした。

_AC-16683 - [GitHub コードの貢献度](https://github.com/magento/magento2/commit/4449d914)_

#### `applyGiftCardToCart` GraphQLの変異からのより記述的なエラー

`applyGiftCardToCart`変異のエラー処理が強化されました。 この突然変異により、期限切れやゼロバランスのギフトカードなどのケースに対して、特定のエラーメッセージが返されるようになり、買い物客の体験が向上しました。 さらに、バックエンドでは、すでに無料となっている注文に追加のギフトカードを適用することを防ぐことができます。

1回の呼び出しでウィッシュリストからすべての項目を削除する`clearWishlist`の変異を追加しました。

_LYNX-786_

#### ギフトオプションは、ゲストカートと顧客カートを組み合わせると、一貫して結合されます

カートの結合中に処理される強化されたギフトオプションにより、より一貫性のあるユーザーエクスペリエンスを実現します。 ギフトオプションは、優先順位付けされた結合ロジックに従うようになり、ゲストユーザーがログインしてカートが既存の顧客カートと結合されたときに意図しない上書きを防ぐことができます。

_LYNX-788_

#### GraphQL `OrderTotal`応答の新しい`grand_total_excl_tax` フィールド

新しいフィールド `OrderTotal.grand_total_excl_tax`がGraphQLの注文応答に追加されました。 このフィールドは、注文の税抜き総計を提供し、APIから直接、税抜き総計にアクセスしやすくします。

利点：

- GraphQLを使用して、任意の注文の税抜き総計を簡単に取得できます。
- 税別の合計を必要とする外部システムとの統合を簡略化します。
- クライアントサイドのカスタム計算の必要性を低減。

#### 新しい管理者設定では、ログイン時にゲストと顧客カートの結合動作が制御されます

ゲストがログインし、既に顧客カートを持っている場合に、カート項目を統合する方法を選択できるようになりました。

- **ゲスト優先度** — ゲストカートの数量を使用します。
- **顧客優先度** – 顧客カートの数量を使用します。
- **Merge Quantities** （default） – 両方の量を合計します。

この動作は、**買い物かごの結合設定**&#x200B;の下の&#x200B;**店舗** > **構成** > **販売** > **チェックアウト**&#x200B;で設定します。 この設定はすべての商品タイプに適用され、サインイン時にゲストカートと顧客カートを組み合わせる方法をマーチャントが完全に制御できます。

#### 支払い方法

- Braintree支払いに対する&#x200B;**ELO カードの種類のサポート**

  Braintree PaymentsでのELO カードタイプのサポートを追加しました。 管理者はクレジットカード設定でELOを有効にし、顧客はチェックアウト時にELO カードで支払うことができます。

  _バンドル–3464_

- **ポーランド人買い物客のBLIK現地支払い方法**

  ポーランドの買い物客のための新しいローカル支払い方法としてBLIKを追加しました。 これにより、既存のBraintree Local Payment Methods （LPM）フロー内で安全な銀行ベースのBLIK支払いが可能になり、ポーランドのお客様のチェックアウトの利便性とコンバージョン率が向上します。

- **請求時に支払う – ドイツの新しいBNPL支払い方法**

  ドイツ人購入者向けの新しいローカル支払い方法として[!DNL Pay Upon Invoice]を追加しました。 [!DNL Pay Upon Invoice]は、PayPalおよびRatepay （「Rechnungskauf mit Ratepay」）によって提供される「BNPL （後払い購入）」オプションです。このオプションを使用すると、顧客はPayPal アカウントを必要とせずに30日以内に商品を受け取り、請求書を支払うことができます。 即時支払いではないため、注文の最終処理はPayPalのサーバーサイド Webhookによって行われます。

  _バンドル–3475_

#### カード保管

- **アカウントエリア経由でGoogle Payを評価する**

  _バンドル–3459_

- **Braintreeのヴォールトカード用のリアルタイムのアカウントアップデータ（RTAU）**

  Braintreeに追加されたリアルタイムアカウントアップデータ（RTAU）機能により、Vault済みのVisa、Mastercard、Discoverのカード詳細は、カードの有効期限が切れるか交換されたときに自動的に更新されます。 これにより、支払い失敗を最小限に抑え、Commerce Vaultを最新の状態に保ち、サポートされていない種類（前払い、Apple Pay、Google Pay）をエラーなくスキップできます。

  _バンドル–3462_

### プラットフォームとインフラ

#### SRI ハッシュストレージのリファクタリング後、マルチテーマストアとマルチロケールストアでのページ読み込みを高速化

SRI ハッシュストレージでは、1つの大きな`sri-hashes.json` ファイルではなく、領域、テーマ、ロケールごとに個別のファイルが生成されるようになりました。 この変更により、各ページに関連するハッシュファイルのみが読み込まれるようになり、ページの読み込み時間が短縮され、複数のテーマまたはロケールを持つストアでのメモリ使用量が削減されます。

_AC-16117_

#### Apache ActiveMQ Artemisは、RabbitMQの長期的な代替品です

Apache ActiveMQ Artemisは、RabbitMQ 4.1に関連するサポート終了のリスクによって駆動される、Adobe Commerceに推奨される長期メッセージブローカーです。 ActiveMQ Artemisは、Commerce リリースライン 2.4.6 ～ 2.4.9で完全にサポートされています。 Adobe Commerce Cloudでは、クラウドネイティブなデプロイメント用にAWS ActiveMQとして提供されます。 キューのコンシューマーとパブリッシャーの両方がSTOMPを使用するように設定できます。


既存のRabbitMQ 4のインストールは、短期的に現在のメッセージキューサービスを使用し続けることを希望するマーチャントに対して引き続き互換性があります。上記の[RabbitMQ 4.2](#add-compatibility-with-rabbitmq-42)との互換性の追加を参照してください。 長期的なサポートのために、ActiveMQ Artemisへの移行を計画します。

管理者ユーザーは、管理者パネルにアクセスするために、マーチャントが有効にしている2FA プロバイダー（Google AuthenticatorやU2Fなど）のうち1つだけを設定する必要があります。 追加の有効なプロバイダーは、必要に応じて後で設定できます。 以前は、複数の2FA プロバイダーが有効になっている場合、すべての管理者ユーザーはログインする前にすべてを設定する必要があり、すべてのプロバイダーにアクセスできないユーザーは摩擦を生じていました。

### 発送

#### USPS統合をRESTful USPS APIに移行する

USPSが発表した従来のWeb Tools APIの廃止に対応するため、Adobe CommerceはUSPS統合を新しいRESTful USPS APIに移行しました。

主な機能強化：

- デュアル API サポート：管理者ユーザーは、従来のWeb ツール APIと新しいRESTful USPS APIの間で、構成設定を介して選択できるようになりました。
- 認証アップグレード：安全なAPI アクセスにOAuth 2.0を使用します。
- データ形式の改善：よりクリーンで効率的な通信のために、XMLの代わりにJSONを使用します。
- 新しい管理者フィールド：
   - ゲートウェイ REST URL （モード：開発またはライブ）
   - クライアント IDと秘密鍵
   - アカウントタイプ、アカウント番号
   - CRID、MID、メーラー識別コード
   - 国際配送のためのAES/ITN
   - REST固有の許可された配送方法

この移行により、Adobe CommerceはUSPS標準を遵守し、システムの信頼性を向上させ、加盟店の出荷統合を将来性のある形で実現します。

_AC-13257_

#### MyDHL RESTful APIへのDHL統合の移行

組み込みのDHL出荷統合は、従来のDHL Express XML APIとの互換性を維持しながら、MyDHL RESTful APIをサポートするようになりました。 加盟店は、Adminで使用するDHL APIを選択でき、既存のXML ベースの設定を損なうことなく、最新のREST機能の恩恵を受けることができます。

_AC-13258_
