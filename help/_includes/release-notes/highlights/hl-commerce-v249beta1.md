# Adobe Commerceのハイライト（v2.4.9-beta1）

## v2.4.9-beta1 のハイライト

Adobe Commerce 2.4.9-beta1 リリースには次のハイライトが適用されます。

### API

#### ストア表示レベルでの REST API プロダクトギャラリー継承コントロール

ストアスコープで REST API を介して製品を更新しても、ペイロードから `media_gallery_entries` が省略されたときや `NULL` に設定されたときに、製品の画像やビデオがグローバルスコープの変更を継承しなくなりました。 また、対応するフィールドを `NULL` に設定することで、REST API を介して製品画像およびビデオのスコープ継承を復元できるようになりました。

_ACP2E-4358 - [GitHub コードの投稿 ](https://github.com/magento/magento2/commit/f7bbcb4e)_

### 管理 UI

#### カタログ価格ルール グリッドのアクション メニュー

Commerce管理のカタログ価格ルール グリッドに、マーチャントが複数のカタログ価格ルールを一度に有効化、無効化または削除できるアクションメニューが含まれるようになりました。 これにより、カタログ価格ルール管理が、買い物かご価格ルールで使用可能な既存の一括アクションと連携するようになり、大きなルールセットの管理に要する時間が大幅に短縮されます。

_AC-13916_

#### コンテンツのステージング用のモバイル表示プレビュー

管理者のステージングプレビュー機能を使用すると、ブラウザーシミュレーションしたモバイルデバイスプレビューを正確にレンダリングして、モバイルデバイスでステージングの更新がどのように表示されるかを視覚的に表現できるようになりました。

_ACP2E-3397 - [GitHub コードの投稿 ](https://github.com/magento/magento2/commit/520f9e30)_

### Braintree

#### 高速チェックアウト

- **Google Pay Express の支払いシートでのプロモーション**

  Google Pay Express Pay Sheet は、プロモーションコードとオファーコードをサポートするようになりました。 買い物客は、Commerceの買い物かごプロモーションをGoogle Pay シート内で直接適用、表示、削除でき、エクスプレスチェックアウトのお客様は標準のチェックアウトフローと同じ割引とインセンティブを受けることができます。

  _バンドル–3476_

- **Apple Pay Express の支払いシートでのプロモーション**

  Apple Pay Express Pay Sheet は、プロモーションコードとオファーコードをサポートするようになりました。 買い物客はApple Pay シート内でクーポンを直接適用できるので、エクスプレスチェックアウトユーザーは、標準のチェックアウトフローと同じ割引やキャンペーンのメリットを享受できます。

  _バンドル–3477_

- **Chromeおよび Firefox のApple Pay**

  Apple Pay は、Safari だけでなく、Chromeや Firefox でも使用できるようになりました。 Apple Pay Express が有効になっている場合、サポートされているストアフロントの場所でApple Pay ボタンを使用でき、お客様はiPhoneでコードをスキャンして支払いを完了できます。

  _バンドル–3478_

- **PayPal Express のサーバーサイド配送コールバック**

  PayPal Express の配送コールバックがクライアントサイドからサーバーサイドに移動されました。 これにより、動的な配送方法、リアルタイムのコスト計算、正確な買い物かごレベルの詳細が PayPal モーダルで直接提供され、信頼性が向上し、連絡先モジュールのサポート、アプリスイッチフロー、Venmo Express などの将来の機能の基盤が構築されます。

  _バンドル–3479_

- **PayPal Contact Module for U.S. merchant express checkout**

  米国のマーチャント向けに新しい PayPal 連絡先モジュールが導入されました。 有効にすると、PayPal Express を使用する購入者は、エクスプレスフロー（PDP、ミニカート、カート、チェックアウトエクスプレス）中に、PayPal モーダル内で直接販売者と共有されるメールアドレスと電話番号を表示および更新できます。 選択した連絡先の詳細は、Commerceの注文に保存されます。

  _バンドル–3480_

#### 支払い方法

- **Braintree支払いに対する ELO カードのタイプのサポート**

  Braintree Payments に ELO カードタイプのサポートを追加しました。 管理者はクレジットカード設定で ELO を有効にできるようになりました。お客様はチェックアウト時に ELO カードを使用して注文を正常に行え、Braintreeを通じてシームレスな取引を確実に行うことができます。

  _バンドル–3464_

- **ポーランドの買い物客のための BLIK 現地支払い方法**

  ポーランドの買い物客のための新しいローカル支払い方法として BLIK を追加しました。 これにより、既存のBraintreeの LPM （Local Payment Methods）フロー内で安全な銀行ベースの BLIK 支払いが可能になり、ポーランドのお客様のチェックアウトの利便性とコンバージョンが向上します。

  _バンドル–3481_

- **請求書による支払 – ドイツの新しい BNPL 支払方法**

  新しい現地支払方法「ドイツの購買者向け請求書払い」を追加しました。 Pay On Invoice は、PayPal と Ratepay （「Rechnungskauf mit Ratepay」）を活用した「Pay Now, Pay Later （BNPL）」オプションであり、PayPal アカウントを必要とせずに、お客様が最初に商品を受け取り、30 日以内に請求書を支払うことができます。 即時支払いではないため、注文の確定は PayPal のサーバーサイドの Webhook によって実行されます。

  _バンドル–3475_

#### カードボルト

- **アカウント領域を介したGoogle Pay のヴォールティング**

  BraintreeでGoogle Pay Vault が有効になっている場合、お客様はアカウント領域を通じてGoogle Pay カードをヴォールティングできるようになりました。 保管されたカードは、保管された支払い方法の下に表示され、チェックアウト時に今後の購入に使用でき、顧客が削除できます。 これにより、Cards と PayPal だけでなく、Google Pay にもヴォールティングのサポートが広がります。

  _バンドル–3459_

- **Braintree ボールトカード用のリアルタイムアカウントアップデーター（RTAU）**

  Braintreeに追加されたリアルタイムアカウントアップデーター（RTAU）機能により、カードの有効期限が切れたりカードが交換されたりすると、Visa、Mastercard、Discover の各カードの詳細が自動的に更新されます。 これにより、支払い失敗を最小限に抑え、Commerce Vault を最新の状態に保ったうえで、サポートされていないタイプ（前払い、Apple Pay、Google Pay）は誤りなくスキップできます。

  _バンドル–3462_

#### 管理ツール

- **Commerceの注文をBraintree ポータルにリンクする**

  Braintree管理者の注文の詳細にCommerce ポータルリンクが追加されました。 リンクをクリックすると、関連する取引がBraintree ポータルで（新しいタブで）開き、Commerce注文のマーチャント ID と取引 ID が使用されます。 これにより、両方のシステムに別々にログインすることなく、直接クロスリファレンスが可能になります。

  _バンドル–3461_

#### セキュリティと互換性

- **3D セキュア向けの基地統合コンテンツセキュリティポリシーの更新**

  コンテンツセキュリティポリシー（CSP）が更新され、最新のカーディナル（3-D セキュア）統合要件がサポートされるようになりました。 これにより、3D セキュアフロー中に使用される基数ホストのスクリプト、iframe および関連リソースがすべてブラウザーの CSP で許可され、リクエストのブロックやチャレンジまたは検証エクスペリエンスの破損が防がれます。

  _バンドル–3485_

- **Braintree支払い拡張機能の PHP 8.5 互換**

  Braintree支払い拡張機能は、PHP 8.4 との互換性を保ちながら、PHP 8.5 ランタイムをサポートするように更新されました。

  _バンドル–3493_

### プラットフォームとインフラストラクチャ

#### OpenSearch 3.x サポート

Adobe Commerce 2.4.9-beta1 は、OpenSearch 3.x と完全に互換性があります。この更新により、マーチャントは、OpenSearch 2.x との下位互換性を維持しながら、パフォーマンスの向上、セキュリティ、長期的なサポートのメリットを得ることができます。

_AC-11846_

#### Valkey 8.x のフルサポート

Adobe Commerce 2.4.9-beta1 では、Redis との完全な CLI コマンドパリティを含む、Redis 互換キャッシュバックエンドとしての Valkey 8.x の包括的なサポートが追加されました。 管理者とクラウドの設定オプションが更新され、シームレスな Valkey 設定が可能になりました。 このサポートは、Redis 7.2 のサポート終了とライセンスの変更によって推進され、Commerce リリース 2.4.5 から 2.4.9-beta1 にわたって、マーチャントに Redis に代わる信頼性の高い完全にサポートされた代替手段を提供します。

_AC-14103、AC-14604_

#### Apache ActiveMQ Artemis サポートは、RabbitMQ に代わるものです

RabbitMQ 4 に関連するサポート終了のリスクを背景に、RabbitMQ の戦略的な代替手段として Apache ActiveMQ Artemis のサポートを追加しました。 ActiveMQ Artemis は、Commerce リリースライン 2.4.6～2.4.9-beta1 で、クラウドネイティブデプロイメント用のAWS ActiveMQ を使用した Adobe Commerce Cloud を含めて完全にサポートされるようになり、キューコンシューマーとパブリッシャーに対する STOMP 設定がサポートされるようになりました。 既存の RabbitMQ 4 のインストールは、現在のメッセージキューサービスを引き続き使用したいマーチャント向けに互換性を維持しています。

_AC-14558_

### PHP と Composer

#### PHP 8.5 の互換性

Adobe Commerce 2.4.9-beta1 以降、PHP 8.5 と完全に互換性があります。一方、PHP 8.4 のサポートを維持し、アップグレードのみの場合は PHP 8.3 を使用できます。 この作業により、コアコード、依存関係、およびツールが最新化されます。これにより、マーチャントは PHP 8.4 のサポート終了に先立って、PCI への準拠と長期的なプラットフォームの正常性を維持しながら、新しい PHP バージョンに安全に移行できます。

_AC-15615_

#### PHP 8.2 のサポートを削除

Adobe Commerce 2.4.9-beta1 以降、PHP 8.2 はサポートされなくなりました。 このプラットフォームは PHP 8.3 以降を対象とし、PHP 8.4 および 8.5 上でクリーンかつ確実に動作するようにコアコード、依存関係、およびツールが更新されています。

_AC-15758_

#### Composer 2.9 の互換性の検証

Adobe Commerce 2.4.9-beta1 は、Composer 2.9 を含む Composer 2.x と完全に互換性があります。この整合性により、下位互換性が維持され、最新の Composer リリースを使用するマーチャントおよび開発者の安定したビルドおよびデプロイメント エクスペリエンスが確保されます。

_AC-14481_

#### Symfony Cache コンポーネントは Zend_Cache を置き換えます

_AC-15823_

#### WYSIWYG Editor が TinyMCE から HugeRTE に移行されました

TinyMCE 5 および 6 のサポートが終了し、TinyMCE 7 との互換性のないライセンスが終了したため、Adobe Commerce WYSIWYG エディターはオープンソースの [HugeRTE](https://hugerte.org/) エディターに移行されました。 この移行により、Adobe Commerceはオープンソースのライセンスに引き続き準拠し、既知の TinyMCE 6 の脆弱性を回避し、マーチャントやデベロッパー向けに最新のサポートされている編集機能を提供します。

_AC-14568_

Adobe Commerceは、PHP 8.5 以降の長期にわたる互換性と安定性を確保するために、従来の Laminas MVC に代わるネイティブの MVC 実装を導入しました。この変更により、パフォーマンスが強化され、外部への依存が軽減され、Commerceの将来に備えた基盤が提供されます。

#### シンプルな二要素認証構成

管理者ユーザーは、管理パネルにアクセスするために、マーチャントで有効な 2FA プロバイダー（Google Authenticator や U2F など）の 1 つのみを設定する必要があります。 必要に応じて、有効なプロバイダーを後で設定できます。 以前は、複数の 2FA プロバイダーが有効な場合、すべての管理者ユーザーは、ログインする前に、有効なすべてのプロバイダーを設定する必要があり、すべての要因にアクセスできないユーザーに摩擦を生じさせていました。

### 送料

#### USPS 統合の RESTful USPS API への移行

USPS が発表した従来の Web Tools API の廃止に準拠するために、Adobe Commerceは USPS 統合を新しい RESTful USPS API に移行しました。

主な機能強化：

- デュアル API のサポート：管理者ユーザーは、設定を使用して、従来の Web ツール API と新しい RESTful USPS API のどちらかを選択できるようになりました。
- 認証アップグレード：安全な API アクセスに OAuth 2.0 を使用します。
- データ形式の改善：よりクリーンで効率的な通信のために、XML ではなく JSON を使用します。
- 新しい管理フィールド：
   - ゲートウェイ REST URL （モード：開発またはライブに基づく）
   - クライアント ID と秘密鍵
   - 勘定科目タイプ、勘定科目番号
   - CRID、MID、メーラー識別コード
   - 国際出荷用の AES/ITN
   - REST 固有の許可されている発送方法

この移行により、Adobe Commerceが USPS 標準に引き続き準拠し、システムの信頼性が向上し、マーチャント向けの将来的な配達確認の出荷統合が実現します。

_AC-13257_

#### DHL 統合の MyDHL RESTful API への移行

組み込みの DHL 配送統合は、従来の DHL Express XML API との互換性を維持しながら、MyDHL RESTful API をサポートするようになりました。 マーチャントは、管理で使用する DHL API を選択でき、既存の XML ベースの設定を中断することなく、最新の REST 機能を活用できます。
