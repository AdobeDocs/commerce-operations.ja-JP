# [!DNL Cloud Automation Patching Service (CAPS)]のベストプラクティスガイド

[!DNL Cloud Automation Patching Service] （[!DNL CAPS]）でのパッチの正常かつ安全な操作には、次のベストプラクティスが不可欠です。 このガイドでは、効果的なパッチ操作、環境管理、およびオペレーショナルエクセレンスに関する包括的なベストプラクティスを提供します。

## プリパッチのベストプラクティス

### 環境対応

**ベストプラクティス：** パッチを適用する前に、必ず環境を十分に準備して、運用を成功させ、リスクを最小限に抑えます。

パッチを適用する前に、環境が適切に準備されていることを確認します。

* **Adobe Commerce Cloud アカウント**
   * Adobe Commerce Cloudのアクティブなサブスクリプション
   * 有効なAdobe Commerce ライセンス
   * 設定されたリポジトリアクセス資格情報
   * プロジェクトと環境の権限

* **環境リソース**
   * 一時的なテスト用に利用可能な環境スロット
   * 十分なストレージ、CPU、およびメモリリソース
   * Adobeリポジトリへのネットワークアクセス
   * 同期のための安定した親環境

* **実稼動環境の準備** （実稼動環境のパッチ適用用）
   * メンテナンスモードを有効にできます
   * Cron ジョブは無効にできます
   * 保守ウィンドウ手順の確立
   * 文書化されたロールバック手順
   * 関係者向けコミュニケーションプランの準備完了

## パッチアプリケーションのベストプラクティス

### タイミングとスケジュール

**ベストプラクティス：** トラフィックの少ない期間にパッチ操作をスケジュールし、関係者と調整してビジネスへの影響を最小限に抑えます。

**パッチ アプリケーションの適切なタイミングを選択：**

* **トラフィックの少ない期間**
   * オフピーク時間中にパッチをスケジュール
   * トラフィックの多いイベント時にパッチを適用するのは避けましょう
   * 検証中の潜在的なダウンタイムを計画

* **実稼動環境に関する考慮事項**
   * **メンテナンスウィンドウ** – 計画的なメンテナンスウィンドウ中に実稼動パッチをスケジュール
   * **お客様とのコミュニケーション** - メンテナンスモードと予想されるダウンタイムをお客様に通知します
   * **チーム調整** – すべてのチームメンバーがメンテナンススケジュールを認識していることを確認します
   * **ロールバックの準備** – 必要に応じて、すぐにロールバックできるチームメンバーを用意します

### 監視と検証

**パッチ処理中：**

* **進行状況を監視**
   * 運用状況をリアルタイムで確認する
   * 警告やエラーに注意する
   * 開始後はプロセスを中断しないでください

* **結果を検証**
   * アプリケーションが成功した後、重要な機能をテストする
   * パフォーマンス指標に劣化がないかを確認する
   * セキュリティ対策が維持されていることを確認する

## パッチ後のベストプラクティス

### 検証とテスト

**ベストプラクティス：** システムの安定性と機能を確保するために、包括的なテストと監視を通じて、パッチアプリケーションの成功を常に検証します。

**パッチの適用後：**

* **機能テスト**
   * あらゆる重要なビジネスプロセスをテスト
   * チェックアウトと支払いフローの確認
   * 管理者パネル機能の確認

* **パフォーマンス監視**
   * ページ読み込み時間の監視
   * データベースのパフォーマンスを確認する
   * リソース使用量の急増に注意する

* **セキュリティ検証**
   * セキュリティ機能が動作していることを確認する
   * 新しいセキュリティの脆弱性を確認する
   * 認証と認証のテスト

## 実稼動環境のベストプラクティス

### プリプロダクションのテスト

**ベストプラクティス：**&#x200B;本番環境の設定を反映したプリプロダクション環境で徹底的なテストを行うことなく、パッチを本番環境に直接適用しないでください。

**実稼動デプロイメントの前に常にパッチをテストする：**

* **テスト環境の設定**
   * テストにステージング環境または統合環境を使用する
   * テスト環境が実稼動設定を反映していることを確認します
   * 可能な限り本番環境のようなデータを使用してテストする

* **包括的なテスト**
   * あらゆる重要なビジネスプロセスをテスト
   * チェックアウトと支払いフローの確認
   * 管理者パネル機能の確認
   * カスタム統合のテスト

* **パフォーマンステスト**
   * パッチのパフォーマンスへの影響を監視
   * パフォーマンスが低下していないか確認する
   * リソースを引き続き使用できることを確認する

### リスクの軽減

**本番環境のパッチ適用中にリスクを最小限に抑える：**

* **コミュニケーションプラン**
   * メンテナンスウィンドウを顧客に通知する
   * 関係者に最新情報を提供
   * エスカレーション手順の準備ができている

* **ロールバック戦略**
   * 必要に応じてパッチをすばやく元に戻す方法について説明します
   * チームメンバーがすぐに対応できるようにします
   * ドキュメントのロールバック手順

* **監視とアラート**
   * パッチ後の問題に対する監視の設定
   * 重大な障害に対するアラートを送信
   * パフォーマンス指標を詳細に監視する

## 主なベストプラクティスの概要

### [!DNL CAPS]の成功に関する重要なベストプラクティス

* 本番環境にパッチを適用する前に、必ずプリプロダクションでテストをおこないます
* メンテナンスモードを有効にし、実稼動パッチ操作のcron ジョブを無効にする
* 運用を詳細に監視し、ロールバック手順の準備を整える
* すべてのパッチ操作を文書化し、包括的なレコードを維持します
* 適切な変更管理手順に従い、適切な承認を得る
* 環境を同期し、適切なリソース割り当てを維持する
* 明確なサポート手順を確立して、チームトレーニングを維持する
* パッチ管理プロセスを定期的に確認して改善する
